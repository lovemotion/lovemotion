# LoveMotion.io — Implementation Plan
*Living document. Last updated: 2026-06-12*

> "We don't swipe. We simulate. Your companion meets theirs first. Only when they agree do you meet."

---

## What LoveMotion Is

LoveMotion.io is a **Companion-to-Companion Pre-Connection Simulation Engine** — a headless matching microservice written in Common Lisp (SBCL). It is called by HeyU.com but operates on its own schedule. Open source (Apache 2.0).

It does one thing with extraordinary depth: **simulate whether two Growth Companions are ready to introduce their humans to each other.**

It does not browse profiles. It does not rank attractiveness. It does not optimize for engagement. It optimizes for *readiness* — the convergence of two behavioral trajectories at a moment when both parties can give and receive what they came for.

---

## Architecture in One Sentence

> HeyU observes users → builds Growth Companion vectors → pushes snapshots to LoveMotion → LoveMotion's Lisp rules engine simulates Companion pairs → outputs introductions when both companions agree.

## The Growth Companion Schema

A Growth Companion is a snapshot of who a user is *becoming*. It has three layers:

### Layer 1 — The Vector (pgvector)
A float array (initially 1536 dimensions — standard embedding size). Generated by HeyU from behavioral signals using OpenRouter. Encodes the full semantic shape of the user's growth journey. Used for fast approximate candidate search.

### Layer 2 — Dimensional Signals (explicit, queryable)
Structured fields the rules engine can reason about directly:

```
growth_level          integer        -- 1-7, Judo belt analog
proof_of_work_score   float          -- normalized 0.0–1.0
contribution_score    float          -- giving to others' growth
values_vector         float[]        -- explicit values dimensions
attachment_style      enum           -- secure/anxious/avoidant/disorganized (inferred, not self-reported)
growth_velocity       float          -- rate of level progression
trajectory_direction  float[]        -- unit vector of growth direction
geographic_region     text           -- coarse grain only
lifestyle_axes        jsonb          -- flexible key-value behavioral signals
last_circle_signals   jsonb          -- behavioral data from recent Growth Circles
```

### Layer 3 — Metadata
```
companion_id          uuid           -- LoveMotion-internal ID
heyu_user_ref         text           -- opaque reference (not PII)
snapshot_at           timestamptz    -- when this snapshot was taken
eligible_for_matching boolean        -- proof-of-work gate passed?
match_cooldown_until  timestamptz    -- prevents premature re-introduction
```

**Key design principle:** LoveMotion never stores PII. `heyu_user_ref` is an opaque token HeyU supplies. LoveMotion knows nothing about the actual person — only the companion.

## The Rules Engine

This is the heart of LoveMotion and the reason we chose Lisp.

### Why a Rules Engine?

Matching is not a black box. When two people are introduced, they will eventually ask *why*. HeyU needs to tell them. The rules engine produces not just a score but a **narrative**: "Your companions aligned on growth trajectory, complementary values, and readiness timing."

A neural network cannot explain itself. A rules engine can.

### Design: Forward Chaining Over Companion Pairs

Rules are CLOS objects. Each rule:
- Takes two companion structs (Twin-A, Twin-B)
- Evaluates a condition
- Returns a score contribution (0.0–1.0) or a veto (nil = hard stop)
- Carries a human-readable explanation string

```lisp
;; Illustration of rule structure (actual implementation in src/engine/rules.lisp)
(defrule compatible-trajectories
  :category :growth
  :weight 0.25
  :veto-threshold nil
  :description "Companions growing in compatible directions"
  :evaluate #'trajectory-alignment-score)

(defrule proof-of-work-gate
  :category :gate
  :weight nil              ; gates have no weight — pass or fail
  :veto-threshold 0.4      ; below this, no match regardless of other scores
  :description "Both companions must demonstrate minimum proof of work"
  :evaluate #'pow-gate-check)
```

### Rule Categories

| Category | Weight | Purpose |
|----------|--------|---------|
| `:gate` | veto | Hard prerequisites. Both must pass. |
| `:growth` | 0.25 | Trajectory alignment and growth compatibility |
| `:values` | 0.25 | Values vector similarity (not identical — complementary) |
| `:readiness` | 0.20 | Growth level, cooldown status, recent circle signals |
| `:contribution` | 0.20 | Giving scores — do they make others better? |
| `:practical` | 0.10 | Geographic, lifestyle compatibility |

### The Simulation

```
simulate(companion-a, companion-b) →
  1. Fire all :gate rules → any veto? → reject pair, explain why
  2. Fire all weighted rules → accumulate category scores
  3. Weighted sum → overall compatibility score
  4. Trajectory dot product → direction alignment bonus
  5. Assemble explanation: list of fired rules + contributions
  6. Return: {score, explanation, ready?, simulated-at}
```

The simulation is **stateless and pure** — given the same two companions, it always returns the same result. This is important: it can be re-run, audited, tested, and explained without side effects.

## The Matching Pipeline

LoveMotion owns the schedule. It runs in configurable windows (e.g., nightly at 2 AM).

```
Pipeline run:
  1. SELECT eligible companions (proof-of-work gate passed, not in cooldown)
  2. For each eligible companion:
     a. pgvector ANN search → top-K candidate companions (fast, approximate)
     b. Filter by growth level compatibility window
     c. Filter by geographic region preference
  3. For each candidate pair (A, B):
     a. Check: have they been simulated recently? (simulation cache)
     b. Run simulate(A, B) → score + explanation
     c. If score >= INTRODUCTION_THRESHOLD → queue as potential match
  4. Rank queued matches by score
  5. Apply diversity constraints (no user in >1 introduction per window)
  6. Write to match_results table
  7. Log simulation_log entry for full audit trail
```

**INTRODUCTION_THRESHOLD** is a configurable parameter, not a magic number baked into code. HeyU can tune this over time as it learns what scores actually produce successful connections.

## API Surface

LoveMotion exposes a minimal REST API over HTTPS. Authentication is API key (shared secret with HeyU).

### Inbound (HeyU → LoveMotion)

```
POST /v1/companions
  Body: companion snapshot (see schema above)
  Response: {companion_id, accepted, eligible_for_matching}

PUT /v1/companions/:companion_id
  Body: updated companion snapshot
  Response: {companion_id, updated_at}

DELETE /v1/companions/:companion_id
  Response: {deleted} 
  Note: user data sovereignty — HeyU calls this when user requests deletion
```

### Outbound (HeyU polls LoveMotion)

```
GET /v1/matches?since=<timestamp>
  Response: [{companion_ref_a, companion_ref_b, score, explanation, ready_at}]

GET /v1/matches/:match_id
  Response: full match record with simulation details

GET /v1/health
  Response: {status, scheduler_last_run, companions_eligible, version}
```

### Admin (internal only, bound to localhost)

```
POST /admin/scheduler/run
  Trigger an immediate pipeline run (for testing/admin)

GET /admin/stats
  Pipeline stats, match rates, score distributions
```

## Database Schema (Initial)

```sql
-- Enable pgvector
CREATE EXTENSION IF NOT EXISTS vector;

-- Growth Companion snapshots
CREATE TABLE companions (
  companion_id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  heyu_user_ref         TEXT NOT NULL UNIQUE,
  embedding             vector(1536),           -- behavioral embedding
  growth_level          INTEGER NOT NULL CHECK (growth_level BETWEEN 1 AND 7),
  proof_of_work_score   FLOAT NOT NULL DEFAULT 0.0,
  contribution_score    FLOAT NOT NULL DEFAULT 0.0,
  attachment_style      TEXT,
  growth_velocity       FLOAT DEFAULT 0.0,
  trajectory_direction  vector(32),             -- lower-dim trajectory vector
  geographic_region     TEXT,
  lifestyle_axes        JSONB DEFAULT '{}',
  last_circle_signals   JSONB DEFAULT '{}',
  eligible_for_matching BOOLEAN DEFAULT FALSE,
  match_cooldown_until  TIMESTAMPTZ,
  snapshot_at           TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  created_at            TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at            TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- ANN index for fast candidate generation
CREATE INDEX companions_embedding_idx
  ON companions USING ivfflat (embedding vector_cosine_ops)
  WITH (lists = 100);

-- Match results
CREATE TABLE match_results (
  match_id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  companion_id_a        UUID NOT NULL REFERENCES companions(companion_id),
  companion_id_b        UUID NOT NULL REFERENCES companions(companion_id),
  score                 FLOAT NOT NULL,
  explanation           JSONB NOT NULL,   -- fired rules + contributions
  ready                 BOOLEAN DEFAULT FALSE,
  pipeline_run_id       UUID NOT NULL,
  simulated_at          TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  consumed_by_heyu_at   TIMESTAMPTZ,      -- NULL until HeyU pulls it
  CHECK (companion_id_a < companion_id_b) -- canonical ordering, no duplicates
);

-- Full audit trail per pipeline run
CREATE TABLE simulation_log (
  run_id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  started_at            TIMESTAMPTZ NOT NULL,
  completed_at          TIMESTAMPTZ,
  companions_evaluated  INTEGER DEFAULT 0,
  pairs_simulated       INTEGER DEFAULT 0,
  matches_produced      INTEGER DEFAULT 0,
  config_snapshot       JSONB    -- threshold values at run time
);
```

## What Makes This Different (The Lisp Advantage)

The rules engine is **explainable by design**. When HeyU introduces two people, it can tell them:

> "Your companions aligned strongly on growth trajectory (↑ 0.24), contribution to others (↑ 0.19), and values orientation (↑ 0.21). Both passed the readiness gate. This introduction was 17 days in the making."

No black box. No "the algorithm said so." Rules are data — they can be tuned, versioned, A/B tested, and explained to users. As the community learns what produces successful connections, rules evolve. That evolution is tracked, auditable, and reversible.

The DEFRULE macro is also the hook that makes LoveMotion extensible by the community. When open-sourced, domain experts can propose rules as pull requests — not ML model tweaks, but human-readable logical propositions. That's a different kind of open source contribution.
